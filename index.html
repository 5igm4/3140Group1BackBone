Project One

Overview

This project is about writing Swing and graphics code, and, along the way, you'll subclass classes, implement interfaces, and generally dig in to Java's brand of object-oriented programming. I've given you some skeleton code that provides the infrastucture for a simple graphical game. Think Pong or Asteroids. I'll ask you to implement a game with some basic behaviors, but the details of the game are up to you. But be thinking: how to make your game a two-player game? What if it could be played across a network? That's something we might explore in project 2.

Objectives

Through your work on this project, you will advance your ability to:
  • Work with code provided by others.
  • Write basic graphics and GUI code in Java.
  • Create basic object-oriented designs.

Introduction to The Skeleton

Most of the code in the skeleton is very closely related to what we've seen in the book and lecture. Perhaps the only big new thing is the Swing Timer class, which basically issues ActionPerformed events at a fixed frequency. Look at the code that uses the Timer: who handles those events? How does handling those events relate to the "animation" of the game?

The other slightly interesting thing is the GameObject class. This is intended to represent, at a fairly general level, the behavior of any graphic elements that will appear in the game. Each object occupies a certain part of the gameboard and has some speed in the x and y directions. Notice that the GameObject class doesn't say much about the how object is actually supposed to be drawn on the screen: that's a question for the game itself to decide. You must not alter the GameObject class, with two exceptions: you need to implement the collide() method and the bounce() method. Of course, you may extend GameObject if you need a class with more specialized behavior.

Game Logic

You need to decide what your game is about. Are you shooting things? Catching things? Bouncing balls around? Something nuttier? Once you've decided on that, you can think about your object-oriented design: for example, do you need to subclass GameObject?

You also need to decide how your game will be controlled. Mouse? Keyboard? What events are the relevant ones for your game? Which object(s) need to listen for these events? Should it be the GamePanel, or a particular GameObject, or...?

When is the Game Over? (Or is it never over...)

Minimum Requirements

So, your game needs to have the following minimum characteristics:
  1. It responds to user controls in an appropriate way. This can be iether mouse or keyboard control, depending on the needs of your game
  2. You must implement the collide() method and use this to affect the game logic—in almost every game of this sort, the collision of two objects is significant (an object got shot, or caught, or bounced off something, or whatever)
  3. You must write a class that either extends or implements.

Other Design Issues

As you're working, keep your eyes open for ways to improve the design of your code. Make sure objects are responsible for their own behavior, make sure you're taking advantage of the API, where you can. And make sure the logic of GamePanel is well-written and efficient (you know how I hate duplicate code). What if your game could have many GameObjects in play at once? Is there a sequence of steps you find yourself always going through, one that might be best put in a helper method?
